## Issue 1 — Cannot open real SQLite databases (CREATE TABLE parse failure)

**Symptom:** `zig build run -- manual.db` exited immediately with `error.UnexpectedToken` while loading table metadata from `sqlite_master`. Multi-column `CREATE TABLE` DDL in the file (e.g. `CREATE TABLE users (id INTEGER, name TEXT)`) could not be parsed.

**Root cause:** In `parse_create_table`, the column separator loop checked for `Token.Semicolon` instead of `Token.Comma` (line 115 of `parser.zig`). After the first column definition the parser saw `,` and failed expecting `)`.

**Fix:** Change the loop to advance on `Token.Comma`, matching `parse_column_list` and `parse_literal_list`.

```zig
// parser.zig — parse_create_table
while (self.nextTokenIs(Token.Comma)) {
    self.advance();
    try column_defs.append(alloc, try self.parse_column_def());
}
```

Added parser test: `parse create table with multiple columns`.

## Issue 2 — Memory leak when CREATE TABLE parse fails

**Symptom:** Debug build reported token / `ArrayList` leaks when `from_file` failed during metadata load (same repro as Issue 1).

**Root cause:** `parse_create_statement` tokenized input but had no `errdefer` to free tokens (or partially built column list) on failure. `parse_statement` already had this pattern; `parse_create_statement` did not.

**Fix:**

- Add `errdefer` on the token slice in `parse_create_statement` (same shape as `parse_statement`).
- Add `errdefer column_defs.deinit(alloc)` inside `parse_create_table`.

## Issue 3 — CLI crashes on query errors

**Symptom:** Invalid SQL or binder errors printed a full stack trace and terminated the process instead of returning to the prompt.

**Root cause:** `cli` called `try eval_query(...)` with no error handling.

**Fix:** Catch errors in the REPL loop and print a short message:

```zig
eval_query(dba, input, alloc) catch |err| {
    std.debug.print("error: {s}\n", .{@errorName(err)});
};
```

## Issue 5 — SELECT fails after INSERT (`BufferExhausted`)

**Symptom:** In the same CLI session, `INSERT` succeeded and data persisted to disk (readable with `sqlite3`), but the next `SELECT * FROM …` returned `error: BufferExhausted`. Reproduced on macOS-created files where the DB header has **12 reserved bytes per page** (`page_size=4096`, `usable=4084`).

**Root cause:** `pager_manager.load_page` passed `rawbuf[0..usable_page_size()]` into `parse_page`. Inserts write a full `page_size` slot via `encode_leaf_page`; cell bodies live at the high end of the page. Truncating the buffer by 12 bytes made cell pointer targets fall past the end of the slice → varint / decoder `BufferExhausted`.

**Fix:** Parse the full on-disk page slot:

```zig
// pager_manager.zig — load_page
return Page.parse_page(
    self.alloc,
    rawbuf[0..self.page_size],
    n,
    &self.header,
);
```

## Issue 6 — Build failure: `Plan.deinit()` missing allocator (pre-session)

**Symptom:** `zig build run` failed to compile: `member function expected 1 argument(s), found 0` at `defer oper.deinit()` in `main.zig`.

**Root cause:** `Plan.deinit` was updated to take `(alloc: Allocator)`; CLI was not updated.

**Fix:** `defer oper.deinit(alloc);` in `eval_query`.

---

## Issue 7 — Build failure: tripwire circular error set (pre-session)

**Symptom:** Release exe build failed in `tripwire.zig` with “function `encode_record` uses its own inferred error set” when `enabled = false` (non-test builds).

**Root cause:** `tripwire.module(..., @TypeOf(encode_record))` created a circular dependency between `encode_record`'s inferred errors and `tw.check` inside that function.

**Fix:** Pass an explicit error set to the tripwire module:

```zig
const tw = @import("tripwire").module(enum {
    after_record_buffer_alloc,
}, error{OutOfMemory});
```
